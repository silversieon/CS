# 📌 6장 : 아키텍처 설계

> 아키텍처 설계는 어떻게 소프트웨어 시스템이 구성되어야 하는지와 시스템 전체 구조 설계를 이해하는 것이다.
>

개별 프로그램, 컴포넌트 단위의 작은 수준의 아키텍처부터, 다른 시스템들과 프로그램 컴포넌트를 포함하는 큰 수준의 아키텍처까지 모두 포함한다.

시스템의 기능적, 비기능적 요구사항을 만족시키기 위한 시스템 구조를 설계하는 프로세스이며, 특정 아키텍처 패턴 또는 스타일에 기반을 둔다.

주로 비기능적 특성과 밀접한 관계가 있기 때문에 스타일과 패턴은 이에 좌우된다.

ex) 성능, 보안성, 안정성, 가용성, 유지보수성 등..(암기X)

## 아키텍처 뷰(아키텍처를 바라보는 관점)

- 논리적 뷰: 객체 또는 객체 클래스로 시스템의 핵심 추상화 ⇒ (클래스 다이어그램)
- 프로세스 뷰: 런타임에 상호작용하는 프로세스들로 구상 ⇒ (시퀀스 다이어그램, 액티비티 다이어그램)
- 개발 뷰: 개발 팀에서 구현한 컴포넌트들로 나눈 것
- 물리적 뷰: 하드웨어와 소프트웨어 컴포넌트들이 어떻게 분산되는지 “배치”를 보여줌
- 유스케이스 뷰: 시스템의 행동(시나리오)를 액터 관점에서 보여줌 ⇒ (유스케이스 다이어그램)

## 아키텍처 패턴

> 서로 다른 시스템과 환경에서 시도되고 시험된 바람직한 사례를 양식화하고 추상화한 기술
>

### MVC 패턴

![image.png](attachment:36e1161d-206f-4e51-850c-c915b1811205:image.png)

> 시스템 데이터로부터 표현과 상호작용을 분리시킨 패턴.
>
- 서로 상호작용하는 세 개의 논리적 컴포넌트들로 구조화된다.
- 뷰는 표현, 제어기는 상호작용, 모델은 데이터를 관리한다.

### 계층 아키텍처 패턴

![image.png](attachment:59dec71e-88b9-4ba5-ad67-fff35221780b:image.png)

![image.png](attachment:6e9fdb1b-26ee-4916-811e-f9650654c1bb:image.png)

> 시스템 계층들로 구성하고, 각 계층은 관련된 기능을 수행한다.
>
- 상위 계층에 서비스를 제공하는데 가장 아래 계층은 시스템 전체에서 사용되는 핵심 서비스 계층.
- 새로운 기능을 기존 시스템 위에 구축할 때 사용한다.
- 단점으로는 현실적으로 명확하게 분리하는 것이 어렵다.

### 저장소 아키텍처 패턴

![image.png](attachment:8a3232e2-715c-4d6c-9538-88d3be0db69b:image.png)

![image.png](attachment:ac1ebf7f-6447-4f51-a8f0-934c510ab111:image.png)

> 시스템 컴포넌트들이 접근할 수 있는 중앙 저장소에서 모든 데이터가 관리된다.(직접 상호작용하지 않고 저장소를 통해 상호작용한다.)
>
- 장기간 저장되어야하는 대량의 정보를 생성하는 시스템에서 사용한다.
- 저장소가 단일 장애점이므로 저장소의 문제가 전체 시스템에 영향을 끼친다.

### 클라이언트-서버 아키텍처 패턴

![image.png](attachment:a28f5d4b-76df-4823-b52e-331baa55fea6:image.png)

> 각 서비스가 독립적인 서버에 의해 제공되는 서비스들의 집합으로, 런타임시 구성을 보여준다.
>
- 공유 데이터베이스에 있는 데이터를 여러 곳에서 접근해야할 때 사용한다.
- 각 서비스가 단일 장애점이며, 네트워크에 영향을 받는다.

### 파이프 필터 아키텍처 패턴

![image.png](attachment:f7ce935d-929d-41ed-a50f-7dc23cc920f8:image.png)

> 기능적 변환들이 입력과 출력을 처리하는 것을 보여주는 시스템 런타임 구성 모델
>
- 순차적, 병렬적 또는 일괄적으로 처리될 수 있으며 데이터 처리 흐름을 보여준다.
- 입력으로부터 출력을 생성하기 위해 단계를 거치는 어플리케이션에서 사용한다.(전송 형식이 일치해야함)
