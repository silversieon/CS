# 📌 8장 : 소프트웨어 테스팅

> 프로그램이 의도된대로 수행되는지 보여주고, 프로그램을 사용하기 전에 결함을 발견하는 것.
>
1. Validation Testing: 예상된 사용을 반영하는 테스트 케이스 집합을 이용하여 정확하게 수행되는 것을 기대
2. Defect Testing: 테스트 케이스가 결함을 드러낼 수 있도록 설계.

테스팅은 폭넓은 소트프웨어 V&V의 일부이다.

Verification: 요구사항에 맞게 만들고 있는가? (기능적/비기능적 명세에 맞게)

Validation: 고객의 기대에 맞게 만들고 있는가? (명세를 넘어서)

### 인스펙션 및 리뷰

실행 시킬 필요 없은 정적 V&V 기술로, 시스템 요구사항, 설계모델, 소스코드 등을 분석하고 검사한다.

> 장점(테스팅에 비해서)
>
1. 테스팅 중에 오류는 다른 오류를 가릴 수 있으므로, 인스펙션을 통해 여러 오류를 한꺼번에 발견할 수 있다.
2. 추가 비용없이 불완전한 버전이 검사될 수 있다.(테스트 개발은 비용이 든다는 점)
3. 비효율성, 부적절한 알고리즘, 서툰 방식 등을 찾아낼 수 있다.

그러나 인스펙션은 소프트웨어 테스팅을 대체할 수는 없다.

### 테스트 케이스

테스트 케이스는 아래 두 사항을 만족시켜야 한다.

1. 테스트에 대한 입력과 시스템으로부터 예상되는 출력의 명세
2. 무엇이 테스트되고 있는지에 대한 서술

### 테스트 주도 개발

테스팅과 코드 개발을 중첩시키는 프로그램 개발 접근법.

프로그래머가 코드의 일부가 실제로 무엇을 하는 것인지에 대한 생각을 분명히하게 도와준다.

> 흐름
>

요구되는 기능 증분 식별 → 테스트 제작, 자동화 구현 → 회귀 테스팅 진행 → 기능 구현(또는 리팩토링) 후 다시 테스팅

> 장점
>
- 코드 커버리지: 작성된 모든 코드의 일부는 적어도 하나의 연관 테스트가 있어야하므로, 개발 프로세스 초기에 결함 발견 가능
- 회귀 테스팅(regression testing): 새로운 테스트가 작성될 때, 기존의 테스트를 포함하여 전체적으로 테스팅함으로써 새로운 버그를 확인한다.
- 단순화 디버깅: 디버깅 도구를 사용할 필요가 없다(테스트 단계에서 바로 찾기 가능)
- 시스템 문서화: 테스트 자체가 코드가 무엇을 하는지 문서화 역할도 한다.

가장 중요한 장점은 **“회귀 테스팅 비용을 감소시키는 것”**
